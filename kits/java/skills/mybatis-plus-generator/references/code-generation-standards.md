# Code Generation Standards

This document details the standards and requirements for code generated by MyBatis-Plus Generator.

## Comment Standards

**IMPORTANT: Generated code must include intelligent, context-aware comments, not just template placeholders.**

### 1. Class Comments

**Requirements:**
- Must explain the purpose of the class based on the table's business context
- Must include table mapping information
- Must list key fields and their purposes
- Must follow Java programming convention format (using `<p>` tags)

**Example:**

```java
/**
 * <p>User entity class</p>
 * 
 * <p>Corresponds to the users table in the database, used to store basic user information.
 * This entity class uses MyBatis-Plus annotations for ORM mapping, supporting automatic table creation and field mapping.</p>
 * 
 * <p>Key fields:
 * <ul>
 *   <li>id: User primary key, auto-increment</li>
 *   <li>username: Username, unique, used for login</li>
 *   <li>email: Email address, unique, used for login and password recovery</li>
 *   <li>password: Encrypted password</li>
 *   <li>status: User status (0-disabled, 1-enabled)</li>
 * </ul>
 * </p>
 * 
 * @author System
 * @since 1.0.0
 */
```

### 2. Method Comments

**Requirements:**
- Must explain the method's purpose based on business requirements
- Must include the type and description of all parameters
- Must include the type and description of the return value
- Must include exception declarations (with full package name)
- Must explain business logic, not just technical operations

**Example:**

```java
/**
 * <p>Find user by email</p>
 * 
 * <p>Query user information by email address, used for user login and email verification scenarios.
 * If the user does not exist, returns null.</p>
 * 
 * @param email java.lang.String User email address, cannot be empty
 * @return com.example.entity.User User entity object, returns null if not found
 * @exception java.lang.IllegalArgumentException Thrown when email address is empty
 */
User findByEmail(String email);
```

### 3. Field Comments

**Requirements:**
- Must explain the field meaning based on business context
- Must include data type and constraints
- Must explain relationships with other fields (if applicable)
- Must not just copy the column name

**Example:**

```java
/**
 * <p>User email address</p>
 * 
 * <p>The unique email address of the user, used for login, password recovery, and receiving notifications.
 * Must conform to email format specifications and be unique across the entire system.</p>
 */
@TableField("email")
private String email;
```

## Template Usage Standards

### 1. Template Selection

Select templates based on the following conditions:
- Object type (Entity, Mapper, Service, etc.)
- Programming language (Java or Kotlin)
- Architecture pattern (MVC, DDD, etc.)

### 2. Template Enhancement

Use templates as a base, but enhance them with:
- Intelligent comments based on table structure and requirements
- Appropriate annotations based on configuration
- Method skeletons for custom methods
- Business logic hints

### 3. Template Variables

Common template variables:
- `${packageName}` - Package name
- `${className}` - Class name
- `${tableName}` - Table name
- `${author}` - Author name
- `${fields}` - Field definitions
- `${methods}` - Method definitions
- `${swagger}` - Whether to enable API documentation (boolean)

For detailed variable list, see: `template-variables.md`

### 4. Swagger Annotations

When API documentation is enabled, templates automatically use OpenAPI 3 annotations:
- `io.swagger.v3.oas.annotations.*`

For detailed usage, see: `swagger-annotations-guide.md`

## Code Generation Order

**CRITICAL: Follow this order when generating code:**

1. **Entity** - First (base for all other objects)
2. **Mapper** - Second (data access layer)
3. **Service** - Third (business interface)
4. **ServiceImpl** - Fourth (business implementation)
5. **Controller** - Fifth (API layer)
6. **DTO/VO/BO** - Sixth (if needed by architecture)

## Custom Method Generation

### 1. Method Signature Generation

Generate method signatures based on user requirements:
- Analyze requirements to determine method name
- Determine parameter types and names
- Determine return type
- Add appropriate annotations

### 2. Method Comment Generation

Generate detailed comments for each custom method:
- Explain method purpose and business logic
- List all parameters and their types
- Describe return value
- List possible exceptions

### 3. Method Skeleton Generation

Generate method skeletons, including:
- TODO comments hinting at implementation details
- Parameter validation hints
- Business logic hints
- Return value handling hints

**Example:**

```java
/**
 * <p>Find user by email</p>
 * 
 * <p>Query user information by email address, used for user login and email verification scenarios.</p>
 * 
 * @param email java.lang.String User email address, cannot be empty
 * @return com.example.entity.User User entity object, returns null if not found
 */
public User findByEmail(String email) {
    // TODO: Implement logic for finding user by email
    // 1. Validate email parameter is not empty
    // 2. Call Mapper to query user
    // 3. Return query result
    return null;
}
```

## Code Quality Requirements

### 1. Comment Completeness

- All classes have complete JavaDoc comments
- All methods have parameter and return value descriptions
- All fields have business meaning comments
- Custom methods have business logic descriptions

### 2. Code Conventions

- Complies with Java programming conventions
- Uses appropriate annotations (Lombok, Swagger, Validation)
- Follows naming conventions (camelCase, PascalCase)
- Appropriate access modifiers

### 3. Business Logic

- Comments reflect business meaning, not just technical descriptions
- Method names clearly express business intent
- Parameters and return values meet business requirements
