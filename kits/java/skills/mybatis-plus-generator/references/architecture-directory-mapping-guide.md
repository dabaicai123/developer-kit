# Architecture Directory Mapping Guide

This document provides a detailed directory mapping guide for code generated by MyBatis-Plus Generator under different architecture types.

## Overview

Depending on the chosen architecture type, generated code needs to be placed in different directory structures. This guide details the directory mapping rules for 5 common architectures.

## 1. Traditional MVC (Model-View-Controller)

**Directory Structure:**
```
src/main/java/{package}/
├── entity/          # Entity classes
├── mapper/           # Mapper interfaces
├── service/          # Service interfaces
├── service/impl/     # ServiceImpl implementation classes
├── controller/       # Controller classes
└── dto/              # DTO (optional)
```

**Mapping:**
- Entity → `{package}/entity/`
- Mapper → `{package}/mapper/`
- Service → `{package}/service/`
- ServiceImpl → `{package}/service/impl/`
- Controller → `{package}/controller/`
- DTO → `{package}/dto/` (if needed)

## 2. DDD (Domain-Driven Design)

**Directory Structure:**
```
src/main/java/{package}/
├── domain/                          # Domain layer
│   ├── model/                       # Domain model
│   │   ├── aggregate/               # Aggregates
│   │   │   └── {entity}/            # Aggregate root (Entity)
│   │   └── valueobject/            # Value objects
│   ├── repository/                  # Repository interfaces (Mapper)
│   └── service/                      # Domain services
├── application/                      # Application layer
│   ├── service/                     # Application services (Service)
│   │   └── impl/                    # Application service implementations (ServiceImpl)
│   └── dto/                         # Application layer DTOs
├── interfaces/                       # Interface layer
│   ├── web/                         # Web interfaces
│   │   ├── controller/              # Controller
│   │   └── dto/                     # Web DTOs
│   │       ├── request/             # Request DTOs
│   │       └── response/            # Response DTOs (VO)
│   └── assembler/                   # DTO assemblers
└── infrastructure/                  # Infrastructure layer
    └── persistence/                  # Persistence implementation
        ├── repository/               # Repository implementations (Mapper implementations)
        └── entity/                   # Persistence entities (Entity)
```

**Mapping:**
- Entity (Domain Model) → `{package}/domain/model/aggregate/{entity}/` or `{package}/domain/model/entity/`
- Mapper (Repository Interface) → `{package}/domain/repository/`
- Service (Application Service) → `{package}/application/service/`
- ServiceImpl → `{package}/application/service/impl/`
- Controller → `{package}/interfaces/web/controller/`
- DTO (Request) → `{package}/interfaces/web/dto/request/`
- VO (Response) → `{package}/interfaces/web/dto/response/`
- BO (Application DTO) → `{package}/application/dto/`

**Note**: In DDD architecture, Entity is a domain model (aggregate root), not a persistence entity. The persistence entity should be placed in `infrastructure/persistence/entity/`.

## 3. Hexagonal Architecture (Ports and Adapters)

**Directory Structure:**
```
src/main/java/{package}/
├── application/                      # Application layer
│   ├── ports/                        # Port definitions
│   │   ├── inbound/                 # Inbound ports (Service interfaces)
│   │   └── outbound/                 # Outbound ports (Repository interfaces)
│   ├── services/                     # Application service implementations (ServiceImpl)
│   └── usecases/                     # Use cases
├── domain/                           # Domain layer
│   ├── model/                        # Domain model (Entity)
│   │   ├── entity/                   # Entities
│   │   └── valueobject/             # Value objects
│   └── service/                      # Domain services
└── infrastructure/                   # Infrastructure layer
    ├── adapter/                      # Adapters
    │   ├── inbound/                  # Inbound adapters
    │   │   └── web/                  # Web adapter
    │   │       ├── controller/       # Controller
    │   │       └── dto/              # Web DTOs
    │   └── outbound/                  # Outbound adapters
    │       ├── persistence/          # Persistence adapter
    │       │   ├── repositoryimpl/   # Repository implementations (Mapper implementations)
    │       │   ├── mapper/           # MyBatis Mapper
    │       │   └── entity/           # Persistence entities
    │       └── external/             # External service adapters
```

**Mapping:**
- Entity (Domain Model) → `{package}/domain/model/entity/`
- Mapper (Repository Interface) → `{package}/application/ports/outbound/`
- Service (Application Port) → `{package}/application/ports/inbound/`
- ServiceImpl → `{package}/application/services/`
- Controller → `{package}/infrastructure/adapter/inbound/web/controller/`
- DTO → `{package}/infrastructure/adapter/inbound/web/dto/`
- Persistence Entity → `{package}/infrastructure/adapter/outbound/persistence/entity/`
- Repository Implementation → `{package}/infrastructure/adapter/outbound/persistence/repositoryimpl/`

## 4. Clean Architecture

**Directory Structure:**
```
src/main/java/{package}/
├── domain/                           # Domain layer (innermost)
│   ├── entity/                       # Business entities (Entity)
│   ├── valueobject/                  # Value objects
│   ├── repository/                   # Repository interfaces (Mapper interfaces)
│   └── service/                      # Domain services
├── application/                      # Application layer
│   ├── usecase/                      # Use cases (Service)
│   │   └── {entity}/                 # Grouped by entity
│   ├── ports/                        # Ports
│   │   ├── input/                    # Input ports
│   │   └── output/                   # Output ports (Repository interfaces)
│   ├── service/                      # Application services (ServiceImpl)
│   └── dto/                          # Application layer DTOs
└── infrastructure/                   # Infrastructure layer (outermost)
    ├── persistence/                  # Persistence implementation
    │   ├── repository/               # Repository implementations (Mapper implementations)
    │   ├── mapper/                   # MyBatis Mapper
    │   └── entity/                   # Persistence entities
    └── web/                          # Web layer implementation
        ├── controller/               # Controller
        └── dto/                      # Web DTOs
```

**Mapping:**
- Entity (Domain Entity) → `{package}/domain/entity/`
- Mapper (Repository Interface) → `{package}/application/ports/output/` or `{package}/domain/repository/`
- Service (Use Case) → `{package}/application/usecase/{entity}/`
- ServiceImpl → `{package}/application/service/`
- Controller → `{package}/infrastructure/web/controller/`
- DTO → `{package}/infrastructure/web/dto/` or `{package}/application/dto/`
- Persistence Entity → `{package}/infrastructure/persistence/entity/`
- Repository Implementation → `{package}/infrastructure/persistence/repository/`

## 5. COLA V5 Architecture

> **Authoritative Reference**: Consistent with the `ddd-cola` skill, following COLA V5 official naming conventions.

**Directory Structure:**
```
src/main/java/{package}/
├── adapter/                          # Adapter layer (aligned with ddd-cola)
│   ├── controller/                   # HTTP/RPC/message inbound handlers
│   │   └── UserController.java
│   ├── scheduler/                    # Scheduled tasks
│   └── dto/                          # Web DTOs (Request/Response)
│       ├── UserCreateRequest.java
│       └── UserResponse.java
├── app/                              # Application layer (COLA V5 uses app, not application)
│   ├── executor/                     # Executors (CQRS)
│   │   ├── command/                  # Command executors
│   │   │   └── UserCreateCmdExe.java
│   │   └── query/                    # Query executors
│   │       └── UserGetQryExe.java
│   ├── service/                      # Application services (Service interfaces)
│   │   └── UserAppService.java
│   └── service/impl/                 # Application service implementations (ServiceImpl)
│       └── UserAppServiceImpl.java
├── domain/                           # Domain layer
│   ├── model/                        # Domain model
│   │   ├── entity/                   # Entities (Entity)
│   │   │   └── User.java
│   │   └── valueobject/             # Value objects
│   ├── gateway/                      # Domain gateway interfaces (COLA uses gateway, not repository)
│   │   └── UserGateway.java
│   ├── service/                      # Domain services
│   └── ability/                      # Domain abilities
└── infrastructure/                   # Infrastructure layer
    ├── persistence/                  # Persistence implementation
    │   ├── mapper/                   # MyBatis Mapper implementations
    │   │   └── UserMapper.java
    │   └── entity/                   # Persistence entities (DO)
    │       └── UserDO.java
    ├── external/                     # External API clients
    └── config/                       # Spring configuration
```

**Mapping:**
- Entity (Domain Model) → `{package}/domain/model/entity/`
- Mapper (Gateway Interface) → `{package}/domain/gateway/`
- Mapper (MyBatis Impl) → `{package}/infrastructure/persistence/mapper/`
- Service (App Service) → `{package}/app/service/`
- ServiceImpl → `{package}/app/service/impl/`
- Controller → `{package}/adapter/controller/`
- DTO (Request) → `{package}/adapter/dto/`
- VO (Response) → `{package}/adapter/dto/`
- BO (App Model) → `{package}/app/executor/` or `{package}/app/model/`
- Command Executor → `{package}/app/executor/command/{entity}/`
- Query Executor → `{package}/app/executor/query/{entity}/`
- Cmd (Command Object) → `{package}/app/executor/command/{entity}/`
- Qry (Query Object) → `{package}/app/executor/query/{entity}/`

**COLA V5 Naming Conventions**:
- Application layer package name is `app` (not `application`)
- Repository interface is named `gateway` (not `repository`)
- Controllers are placed in `adapter/controller/` (not `adapter/web/controller/`)
- Uses `Cmd` suffix for command objects, `Qry` suffix for query objects
- Uses `Exe` suffix for executors

## How to Determine Output Directory

### Step 1: Confirm Base Package Path

Ask the user:
```
Please provide the project's base package path (e.g., com.example.order)
```

### Step 2: Build Path Based on Architecture Type

Based on the architecture type and base package path, build the complete file path:

**Example:**
- Architecture: DDD
- Base package: `com.example.order`
- Table name: `user`
- Entity path: `src/main/java/com/example/order/domain/model/aggregate/user/User.java`

### Step 3: Verify Directory Exists

Before generating code, check if the directory exists:
- If the directory does not exist, create it
- If the directory already exists, confirm whether to overwrite existing files

### Step 4: Generate Files

Generate files in the correct location.

## Complete Path Examples

Assuming base package path is `com.example.order` and table name is `user`:

- **MVC**: 
  - Entity: `src/main/java/com/example/order/entity/User.java`
  - Controller: `src/main/java/com/example/order/controller/UserController.java`

- **DDD**: 
  - Entity: `src/main/java/com/example/order/domain/model/aggregate/user/User.java`
  - Controller: `src/main/java/com/example/order/interfaces/web/controller/UserController.java`

- **Hexagonal**: 
  - Entity: `src/main/java/com/example/order/domain/model/entity/User.java`
  - Controller: `src/main/java/com/example/order/infrastructure/adapter/inbound/web/controller/UserController.java`

- **Clean**: 
  - Entity: `src/main/java/com/example/order/domain/entity/User.java`
  - Controller: `src/main/java/com/example/order/infrastructure/web/controller/UserController.java`

- **COLA** (aligned with ddd-cola skill):
  - Entity: `src/main/java/com/example/order/domain/model/entity/User.java`
  - Gateway: `src/main/java/com/example/order/domain/gateway/UserGateway.java`
  - AppService: `src/main/java/com/example/order/app/service/UserAppService.java`
  - Controller: `src/main/java/com/example/order/adapter/controller/UserController.java`

## Important Notes

1. **Domain Entity vs Persistence Entity**:
   - DDD, Hexagonal, and Clean architectures require distinguishing these
   - Domain entities contain business logic and are placed in the domain layer
   - Persistence entities are database mappings and are placed in the infrastructure layer

2. **Mapper Interface Location**:
   - In DDD, Hexagonal, and Clean architectures, there are typically two locations:
     - Repository interface (defined in domain layer)
     - MyBatis Mapper (implemented in infrastructure layer)

3. **DTO Classification**:
   - In DDD architecture, Request and Response are stored separately
   - Other architectures may store them in a single directory

4. **Package Path Conversion**:
   - Package paths use dot separators: `com.example.order`
   - File paths use slash separators: `com/example/order/`
   - Base path: `src/main/java/`
